The Project

What is Lingua?

A web app for exploring Catalan phrases and idioms — with dictionary definitions, categories, and full-text search.

  • Word Search: Look up any Catalan word, see definitions from two major dictionaries (DCVB & GDLC) side by side
  • Phrase Browse: Explore expressions organized by categories, with explanations and related phrases
  • Admin Panel: CRUD interface for managing all content

Before Writing Code

LLM-Assisted Decision Making

The stack wasn't chosen by one AI — it evolved through conversations with multiple LLMs.

Gemini: Stack Research

First conversation: described the project goals. Gemini recommended 3 stack options — SvelteKit + PostgreSQL FTS, a dedicated search engine (Meilisearch), or a graph DB (Neo4j). Went with Option 1 for simplicity.

Gemini: PWA + Konsta UI

Second conversation: asked for a minimum PWA example with Konsta UI (mobile-native component library). Got a working scaffold — but later dropped Konsta UI in favor of a simpler custom Tailwind design system, since PWA already handles installability.

Claude: All Implementation

With the stack decided, Claude Code took over for the entire build — from design system to production deployment. Claude also researched within its domain: FTS config for Catalan, Neon driver setup, PostHog integration.

// Decision pipeline: Gemini → "What stack for a Catalan dictionary?" → PostgreSQL FTS + Drizzle + SvelteKit Gemini → "Scaffold PWA + Konsta UI" → Got example, but dropped Konsta UI → Custom Tailwind design system instead Claude → All coding, plans, and implementation → Researched specifics within each step
Using different LLMs for different strengths: Gemini for broad research and options, Claude Code for deep implementation and iteration.

// Typical session flow: 1. Describe the feature I want 2. Claude drafts a plan (.claude/plans/) 3. We refine scope together 4. Claude implements step by step 5. I review, test, and give feedback 6. Claude commits each phase 7. Update CLAUDE.md with learnings

Guardrails for AI

The Design System as a Contract

One of the first things Claude built was a full design system with dedicated pages (/design-system & /sistema-disseny). This wasn't just documentation — it became a constraint system for the AI itself.

Why it matters for AI-driven development

  • Limits creative freedom: Without it, Claude would invent new styles per feature. The design system forces consistency — use bg-brand, not a random hex color
  • Reference for composition: When Claude builds a new page, it checks the design system for how cards, buttons, badges, and forms should look
  • Codified in CLAUDE.md: Conventions like "use semantic aliases, not raw palette" are in the project instructions — so every session follows the same rules
  • Evolved with the project: When a11y audits required better contrast, the design system's brand color shifted from primary-500 to primary-700 — and everything updated automatically

Color Palette

Vermillion/terracotta — primary-50 through primary-900

Typography

Lora for headings

Semantic Aliases

--color-brand: primary-700 /* not a random color */ --color-surface: neutral-50 --color-base: neutral-800 --color-muted: neutral-600

Mobile-First & Beyond

PWA, Icons & Live Browser Testing

PWA from Day One

The app was designed to work on mobile from the start. Instead of a native app wrapper like Konsta UI (initially considered), we went with a pure PWA approach — installable, offline-capable, and lighter to maintain.

Icon Pipeline

Claude built a full icon generation pipeline: a single SVG source → scripts/generate-icons.ts produces all PNG sizes + favicon. Then we wrapped the entire flow (icons + screenshots) into a reusable Claude skill (/refresh-pwa-assets) for one-command regeneration.

Live Testing with MCPs

Claude used a Puppeteer MCP server to launch a real browser, navigate the running app, take screenshots, and verify changes visually — all within the same conversation. This closed the feedback loop: code → run → see → fix.

What are MCPs?

Model Context Protocol servers extend Claude's capabilities. Instead of just reading/writing files, Claude can control a browser, interact with APIs, or use any custom tool — enabling real end-to-end testing during development.

Developer Experience

VS Code Extension vs Terminal

Most of Lingua was built using the VS Code extension. Here's how they compare.

VS Code Extension

Advantages

  • See file changes in real time as Claude edits
  • Easy to review diffs inline before accepting
  • Integrated with your existing editor workflow
  • Can highlight code and ask questions in context
  • Side-by-side: code + Claude chat

Drawbacks

  • Feature releases lag behind the terminal CLI
  • Some commands not available (e.g. // shortcuts)
  • Can feel slower for rapid multi-file edits
  • UI can get cramped on smaller screens

Terminal CLI

Advantages

  • Always has the latest features first
  • Full power: hooks, skills, slash commands
  • Better for long autonomous tasks
  • Lighter on system resources

Drawbacks

  • No real-time file diff preview
  • Have to switch between terminal and editor
  • Harder to point at specific code visually

Developer Experience

The Ralph Wiggum Loop

An autonomous development loop: give Claude a prompt, let it work, and when it finishes — feed it the same prompt again. It sees its own changes and iterates until done.

// The core idea: while (not done) { feed prompt to Claude Claude reads its previous work Claude picks up where it left off Claude commits when a step is complete } // Exits when <promise>TASK COMPLETE</promise>

Used for: Word Metadata Feature

A 6-commit plan — new DB table, FTS triggers, search integration, full admin CRUD, nav updates. The loop ran through all steps, committing after each phase.

What went well

  • Pattern reuse: Admin pages mirrored existing phrases pages — consistent code
  • Proper conventions: CLAUDE.md ensured Svelte 5 runes, design system tokens, FTS patterns
  • Multi-step execution: 6 commits, each matching the plan

Developer Experience

Ralph Loop: What Went Wrong

Skipped testing

Despite the prompt asking to test everything, the first steps were not properly verified before moving on. Had to stop the loop and intervene to ask Claude to actually test what it built.

Permission mode reset

Was running in dangerous mode (auto-approve all). After stopping to fix testing, the mode silently reset. Left the loop running overnight — it got stuck waiting for an edit approval that never came.

Post-loop cleanup

Small UI bugs remained: related words not showing in list, form not resetting after edit, toast position, CI workflow needed updating.

The loop is powerful but not fire-and-forget. Think of it as a junior developer working overnight — you still need to review in the morning.

Lesson: Be explicit in the prompt

Don't say "test everything." Add explicit validation checkpoints per phase in the plan: what to verify, how to verify it, and what success looks like. Claude needs concrete criteria or it will rush to the next step.

Lessons Learned

Key Takeaways

Plans Are Essential

Starting with a plan keeps Claude focused and gives you a shared reference. It also captures deviations and gotchas — invaluable for future sessions.

CLAUDE.md Is Your Memory

The project instructions file (CLAUDE.md) is what makes Claude productive across sessions. Conventions, gotchas, and architecture decisions all live here.

Review Everything

Claude is fast but not infallible. Code review is still necessary — especially for security (auth), edge cases, and making sure the generated code matches your mental model.

Best for Greenfield

Claude shines at scaffolding new features end-to-end. It's less effective when debugging subtle issues in existing code it didn't write.

Domain Knowledge Is Yours

I brought the idea, the language expertise, and the product vision. Claude brought the implementation speed. That combination is powerful.

It Gets Better Over Time

As CLAUDE.md grew and conventions solidified, each session was more productive. Claude made fewer mistakes because it had more context.

What's Next

Future Work: Community Proposals

The next feature opens Lingua to its users — allowing anyone to propose new phrases and word metadata, with admin approval.

Public Side

  • Propose a phrase: Users can suggest new expressions with explanation, category, and related phrases
  • Propose word metadata: Add notes and related words for entries that don't have them yet
  • No account required: Low friction — just fill a form and submit

Admin Side

  • Proposal queue: New section in the admin panel to review pending submissions
  • Approve / Edit / Reject: Admins can accept proposals as-is, modify before publishing, or discard
  • Spam prevention: Rate limiting and basic moderation tools
The goal: make Lingua a living, community-driven resource for Catalan expressions — not just a static dictionary managed by one person.
